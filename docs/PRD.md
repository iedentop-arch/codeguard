## 一、产品概述

### 1.1 背景与目标

当前管理5-20家乙方团队时，代码规范依赖文档传达，执行效果难以追踪。本系统将参考项目的治理体系（50+规范文档、SLA评分、六层质量门禁、交付验收清单）转化为可运行的Web平台，实现乙方入驻→代码审查→交付管理的全流程数字化管控。

### 1.2 核心价值

- **标准化入驻**: 乙方通过规范学习+考试认证后方可提交代码
- **自动化质量门禁**: 六层质量门禁自动检查，减少人工审查负担
- **SLA量化评分**: 7维度评分体系，客观衡量乙方表现
- **交付验收数字化**: 10维度验收清单，确保交付质量

## 四、数据模型

### 4.1 核心实体

```
vendors          - 乙方组织
vendor_members   - 乙方成员
spec_documents   - 规范文档
exam_questions   - 考试题库
exam_attempts    - 考试记录
pull_requests    - PR追踪
quality_gates    - 质量门禁结果
monthly_scores   - SLA月度评分
deliveries       - 交付包
delivery_checklists - 验收清单
```

### 4.2 数据库选型

- **主数据库**: MySQL 8.0
- **ORM**: SQLAlchemy 2.0
- **迁移工具**: Alembic

## 五、技术架构

### 5.1 后端

- **框架**: FastAPI
- **认证**: JWT (python-jose)
- **数据库**: SQLAlchemy + Alembic
- **验证**: Pydantic v2
- **任务队列**: ARQ (Redis)

### 5.2 前端（已完成原型）

- **框架**: React 18 + TypeScript
- **构建**: Vite
- **样式**: Tailwind CSS
- **图表**: Recharts
- **组件**: shadcn/ui

### 5.3 第三方集成

- **GitHub App**: PR Webhook + Checks API
- **对象存储**: 本地文件系统（可扩展S3）

## 七、质量门禁设计

### 7.1 六层门禁

| 层级 | 名称 | 检查项 | 失败影响 |
|------|------|--------|----------|
| L1 | 红线检查 | Security/Lint/Type/UnitTest | 阻断 |
| L2 | 必需检查 | API文档/AI标记/依赖安全 | 阻断 |
| L3 | AI辅助 | 架构/复杂度/重复代码 | 建议 |
| L4 | 指标采集 | Ruff/MyPy/Coverage | 记录 |
| L5 | 文档验证 | API/Agent/ADR | 建议 |
| L6 | 合规审查 | 内容扫描/法规清单 | 阻断 |

## 八、SLA评分体系

### 8.1 七维度权重

| 维度 | 权重 | 红线 | 目标 |
|------|------|------|------|
| CRITICAL违规 | 30% | =0 | =0 |
| WARNING趋势 | 15% | 无增长 | ≥10%下降 |
| 代码质量 | 15% | ≥80 | ≥90 |
| 合规通过率 | 15% | ≥90% | ≥95% |
| PR评审效率 | 10% | ≤4轮 | ≤2轮 |
| AI代码标记率 | 10% | ≥95% | 100% |
| CI成功率 | 5% | ≥85% | ≥95% |

### 8.2 等级划分

| 等级 | 分数范围 | 颜色 | 处理 |
|------|----------|------|------|
| A | ≥90 | 绿色 | 正常合作 |
| B | 75-89 | 蓝色 | 正常合作 |
| C | 60-74 | 黄色 | 改进计划 |
| D | <60 | 红色 | 暂停/终止 |
